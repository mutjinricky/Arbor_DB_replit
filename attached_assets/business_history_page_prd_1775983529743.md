# 기존 리플릿 웹앱에 사업이력 페이지를 추가하기 위한 PRD

## 1. 문서 목적
이 문서는 **기존 리플릿 웹앱에 사업이력 페이지를 신규 추가하거나 기존 프로젝트 페이지를 대체하는 방식으로 통합 구현**하기 위한 제품 요구사항 문서다.

이 문서의 목적은 다음과 같다.
- 기존 웹앱 안에 사업이력 페이지를 일관된 UI/UX로 추가한다.
- 공무원 담당자가 실제 업무에 사용할 수 있는 목록 중심 화면을 제공한다.
- 설계서 파일 업로드를 통한 사업 자동 등록 기능을 포함한다.
- 등록된 사업 데이터가 새로고침 이후에도 유지되도록 한다.

이 문서는 **무엇을 만들어야 하는지**를 정의하는 문서이며, 프로토타입 디버깅 메모나 구현 중 발생한 개별 오류 기록은 본문 범위에 포함하지 않는다.

---

## 2. 프로젝트 배경
기존 웹앱에는 수목관리 관련 기능이 이미 존재한다. 여기에 사업 단위 이력 관리 기능을 추가하여, 공무원 담당자가 과거와 현재의 사업을 한 화면에서 조회하고, 비교하고, 인수인계 및 보고자료 작성에 활용할 수 있도록 해야 한다.

기존 프로젝트 페이지는 일반적인 프로젝트 개념에 가깝지만, 실제 사용자 관점에서는 **사업이력 페이지**로 이해되는 구조가 더 적합하다. 따라서 본 요구사항은 기존 프로젝트 페이지를 사업이력 페이지로 전환하거나, 동일한 역할의 신규 메뉴를 추가하는 방향을 전제로 한다.

---

## 3. 사용자 및 사용 시나리오

### 3.1 주요 사용자
- 지자체 공무원 담당자
- 수목관리, 공원녹지, 산림공원, 병해충 관리 담당자
- 인수인계, 보고서 작성, 민원 대응이 필요한 실무자

### 3.2 주요 사용 시나리오
사용자는 주로 아래와 같은 목적으로 이 페이지를 사용한다.
- 같은 지역에서 과거에 어떤 사업을 했는지 조회
- 올해와 전년도 사업 비교
- 사업명, 시행사, 상태, 공기, 예산 등을 빠르게 확인
- 민원 대응 또는 회의자료 작성 시 근거 자료 확인
- 설계서 파일을 등록하여 새로운 사업 이력을 시스템에 추가
- 담당자 변경 후 기존 사업 흐름 파악

---

## 4. 목표

### 4.1 제품 목표
- 기존 웹앱 안에 자연스럽게 통합되는 사업이력 페이지를 제공한다.
- 목록 중심 행정형 UI를 통해 사업 탐색 효율을 높인다.
- 사업 상세는 팝업 구조로 분리하여 메인 목록의 가독성을 유지한다.
- 파일 업로드 기반 사업 등록 흐름을 제공한다.
- 등록 데이터는 새로고침 이후에도 유지된다.

### 4.2 비목표
아래 항목은 본 문서의 핵심 목표가 아니다.
- 완전한 OCR 시스템 구현
- 외부 전자결재 시스템 연동
- 실제 계약 시스템과의 자동 연계
- GIS 정밀 분석 엔진 구현

---

## 5. 통합 방식 요구사항

### 5.1 기존 웹앱과의 관계
사업이력 페이지는 **기존 리플릿 웹앱의 일부 페이지로 통합**되어야 한다.

중요 원칙:
- 독립적인 새 앱처럼 만들면 안 된다.
- 기존 웹앱의 라우팅, 레이아웃, 메뉴 구조, 공통 스타일을 최대한 재사용해야 한다.
- 기존 디자인 시스템이 있다면 우선적으로 그 체계를 따른다.

### 5.2 페이지 추가 방식
아래 두 방식 중 하나로 구현 가능하다.
1. 기존 `프로젝트` 페이지를 `사업이력` 페이지로 대체
2. 신규 `사업이력` 메뉴/라우트 추가

최종 선택은 기존 웹앱 구조에 맞게 결정하되, 사용자 입장에서는 **사업이력이라는 개념**으로 인지되어야 한다.

### 5.3 라우팅 요구사항
- 기존 웹앱 라우팅 체계에 맞춰 추가한다.
- 상세는 별도 독립 페이지가 아니라 **팝업 기반 상세 보기**를 우선 원칙으로 한다.
- 목록 화면에서 상세 정보가 페이지 하단에 길게 붙는 구조는 사용하지 않는다.

## 6. 핵심 기능 요구사항

## 6.1 상단 요약 영역
페이지 상단에는 현재 데이터 기준으로 핵심 현황을 요약 표시한다.

표시 항목:
- 전체 사업 수
- 진행 중 사업 수
- 완료 사업 수
- 총 사업 예산
- 총 집행액 비율
- 총 집행액 금액

요구사항:
- `총 집행액 비율`은 다른 수치보다 크게 보여야 한다.
- 요약값은 목록 데이터와 동일한 원천에서 계산해야 한다.
- 사업 데이터가 추가되거나 수정되면 요약값도 자동으로 갱신되어야 한다.

## 6.2 사업이력 목록
메인 화면은 카드형보다 **행정형 목록/표 구조**를 사용한다.

필수 컬럼:
- 사업명
- 시행사
- 사업상태
- 공기
- 상세 진입 아이콘

요구사항:
- 첫 진입 시 사업이력목록이 우선적으로 보여야 한다.
- 목록이 길면 내부 스크롤이 가능해야 한다.
- 마우스 드래그로 위아래 이동도 가능해야 한다.

정렬 규칙:
1. 계획중
2. 진행중
3. 완료
4. 같은 상태 내 최신순

## 6.3 검색 및 필터
검색/필터는 별도 큰 블록으로 분리하지 않고 **목록 상단에 통합**한다.

지원 항목:
- 사업명 검색
- 연도 체크박스 필터
- 지역 체크박스 필터
- 사업유형 체크박스 필터
- 사업상태 체크박스 필터
- 필터 열기/닫기
- 필터 초기화
- 활성 필터 칩 표시

요구사항:
- 필터는 기본적으로 접힌 상태를 허용한다.
- 사용자가 페이지에 처음 들어왔을 때 목록이 한눈에 들어와야 한다.
- 체크박스로 여러 조건을 조합할 수 있어야 한다.

## 6.4 사업 상세 팝업
목록의 각 사업을 클릭하면 상세 팝업을 연다.

팝업 상단 카드:
- 사업 상태
- 공기
- 총 예산

상세 블록:
- 사업 지도 보기
- 연결 수목 목록
- 준공사진 탭
- 지정된 계약업체

요구사항:
- 상세는 팝업으로만 보여준다.
- 메인 목록 페이지 하단에 상세를 붙이지 않는다.
- 기존 웹앱 스타일에 맞는 모달 구조를 사용한다.

## 6.5 연결 수목 팝업
사업 상세 내 연결 수목 클릭 시 수목 사진 팝업을 연다.

표시 항목:
- 수목 ID
- 좌표
- 사진

예외 처리:
- 사진이 없으면 가짜 예시 이미지 대신 `사진 없음` 상태를 보여야 한다.

## 6.6 준공사진 탭
사업 상세 팝업 안에 준공사진 탭을 둔다.

동작 요구사항:
- 준공사진이 여러 장이면 탭 또는 버튼으로 전환 가능해야 한다.
- 사진이 없으면 `준공사진 없음` 상태를 보여야 한다.

## 6.7 설계서 파일 업로드 기반 자동 등록
`사업설계서 등록` 버튼을 통해 업로드 팝업을 연다.

지원 형식:
- PDF
- TXT
- MD

동작 순서:
1. 파일 선택
2. 파일 내용 읽기
3. 필요한 사업 정보 자동 추출
4. 우측 미리보기 표시
5. 사용자가 `사업 등록` 클릭
6. 목록 및 상단 요약 자동 갱신

중요 원칙:
- 사업 정보는 수동 텍스트 입력보다 **파일 업로드 기반 등록**을 우선한다.
- 등록 후 목록과 요약 정보가 같은 데이터 원천을 기준으로 동시에 갱신되어야 한다.

---

## 7. 데이터 지속성 요구사항
사업 데이터는 일시적인 화면 상태가 아니라 **지속 저장 대상**이어야 한다.

필수 요구사항:
- 업로드로 등록한 사업은 새로고침 이후에도 유지되어야 한다.
- 목록, 요약, 상세 팝업은 저장된 동일 데이터 원천을 기준으로 다시 복원되어야 한다.
- 메모리 상태만 수정하고 끝나는 방식은 허용하지 않는다.

권장 구현 방향:
- 프로토타입 단계: localStorage 또는 IndexedDB
- 실제 서비스 단계: 백엔드 DB 또는 서버 저장소 연동

## 8. 화면 구조 요구사항

## 8.1 메인 페이지 구조
1. 페이지 헤더
   - 페이지명: 사업이력
   - 액션 버튼: PDF 출력, 엑셀 다운로드, 보고서 생성, 사업설계서 등록
2. 상단 요약 카드 영역
3. 목록 및 검색/필터 통합 영역
4. 목록 테이블

## 8.2 상세 팝업 구조
1. 팝업 헤더
2. 사업명 / 사업 ID / 요약 텍스트
3. 상태 / 공기 / 총 예산 카드
4. 2열 상세 블록
   - 사업 지도 보기
   - 연결 수목
   - 준공사진
   - 지정된 계약업체

## 8.3 설계서 등록 팝업 구조
좌측:
- 파일 업로드 설명
- 드롭존 또는 파일 선택 UI
- 선택 파일명
- 분석 중 상태
- 에러 상태

우측:
- 자동 추출 미리보기
- 사업명
- 연도
- 지역
- 사업유형
- 사업상태
- 공기
- 총예산
- 집행액
- 수목 수
- 취소 / 사업 등록 버튼

## 9. 데이터 모델 요구사항
사업 객체는 최소한 아래 속성을 관리할 수 있어야 한다.

```ts
{
  id: string,
  name: string,
  year: number,
  region: string,
  type: string,
  status: '계획중' | '진행중' | '완료',
  period: string,
  budget: number,
  spent: number,
  vendor: string,
  vendorEmail: string,
  department: string,
  summary: string,
  treeCount: number,
  mapCount: number,
  connectedTrees: Array<{
    id: string,
    coord: string,
    photoUrl: string,
  }>,
  contractors: Array<{
    name: string,
    email: string,
    initial: string,
  }>,
  coords: Array<{
    x: number,
    y: number,
    size: 'sm' | 'lg',
  }>,
  completionPhotos: Array<{
    id: string,
    label: string,
    photoUrl: string,
    date: string,
  }>
}
```

## 10. 설계서 파싱 요구사항
설계서 파일에서 아래 항목을 자동 추출할 수 있어야 한다.

- 사업명
- 연도
- 지역
- 사업유형
- 공기
- 예산
- 시행사
- 수목 수

기본 원칙:
- PDF는 텍스트 레이어 기반 읽기를 우선한다.
- 텍스트는 줄 단위로 정리한 뒤 항목 추출에 사용한다.
- 파일명 fallback은 최후 수단으로만 사용한다.
- 추출 실패 시 시스템이 비정상 문자열을 그대로 화면에 노출하지 않도록 정제해야 한다.

## 11. UX 원칙
반드시 유지해야 하는 원칙은 아래와 같다.

- 목록이 메인이고 상세는 팝업이다.
- 검색/필터는 목록 안에 통합한다.
- 상단 요약은 데이터 변경 시 자동 갱신된다.
- 등록 데이터는 새로고침 후에도 유지된다.
- 사진이 없으면 `사진 없음`을 명시한다.
- 공무원용 행정형 화면으로 구성한다.
- 기존 웹앱의 스타일과 레이아웃을 최대한 재사용한다.

## 12. 구현 완료 기준
아래 조건을 만족하면 1차 구현 완료로 본다.

1. 기존 리플릿 웹앱 안에서 사업이력 페이지 접근 가능
2. 상단 요약 영역 정상 표시
3. 목록, 검색, 체크박스 필터 정상 동작
4. 목록 클릭 시 상세 팝업 정상 동작
5. 연결 수목 팝업 정상 동작
6. 준공사진 탭 정상 동작
7. 설계서 파일 업로드 및 자동 등록 가능
8. 등록 후 목록과 요약 자동 갱신
9. 등록 데이터가 새로고침 후에도 유지
10. 기존 앱 스타일과 이질감 없이 통합

## 13. 다른 AI 또는 개발자에게 전달할 작업 지침
- 이 작업은 새 앱 제작이 아니라 **기존 리플릿 웹앱 내부에 사업이력 페이지를 통합 추가하는 작업**이다.
- 기존 레이아웃, 공통 컴포넌트, 라우팅 구조를 최대한 재사용한다.
- UI는 목록 중심, 상세 팝업 구조를 유지한다.
- 설계서 파일 업로드 기반 등록 기능을 포함한다.
- 저장은 반드시 새로고침 이후에도 유지되도록 구현한다.
- 더미 데이터로 먼저 동작 가능하게 만들되, 추후 실제 저장소 연동이 가능하도록 구조화한다.

## 14. 최종 요약
이 문서는 기존 리플릿 웹앱에 공무원용 **사업이력 페이지**를 통합 추가하기 위한 PRD다.

핵심은 다음 4가지다.
- 목록 중심 사업 조회
- 팝업 기반 상세 확인
- 설계서 파일 업로드 기반 자동 등록
- 등록 데이터의 영속 저장
